**Cycle ID:** 20260703224551
**Type:** Feature
**Created:** 2026-07-03 22:45:51

## Overview

複数リポジトリ（例: 仕事の frontend / backend-admin / backend-api / client / gameserver、社内 GitLab ホスティング）を横断して質問に答えられる wiki を作るための入口。git リポジトリ（URL またはローカルパス）を ingest 可能にする。

clone は自動化する: `ghq` があれば `ghq get --shallow`、なければ `git clone --depth 1` にフォールバック。取り込み時の HEAD commit hash を `source_revision` として記録し、スナップショット時点を追跡可能にする（差分更新は将来）。

**v2 の構造転換（team review 合意）**: 成否を決めるのは clone 自動化ではなく「取り込んだ後、横断質問に耐える記事になるか」。よって (A) 手動 dogfooding → (B) compile 規範の整備 → (C) スクリプト化 → (D) 複数リポジトリ運用、の順に**価値検証を最前段に置く**。QueryLog（Complete 表記・実運用ゼロ）の再来を構造的に防ぐ。

### Goals

- **Phase A**: スクリプトを書く前に、実リポジトリ1つで手動 ingest → 記事化 → query を通し、記事品質・file ingest 経路（**実績ゼロの未検証経路**）・機密スキャン誤検知率を実測する
- **Phase B**: `references/compilation-guide.md` に「repo ソースの compile」節を新設（repo 概要記事の必須構成 / 横断フロー記事 / 出典規約 / 段階的読解プロトコル / untrusted 取り扱い）
- **Phase C**: `repo_ingest.py`（層分割: `lib/domain/repo.py` + `lib/service/repo_clone.py` + 薄い CLI）で clone 自動化 + manifest 生成。複数リポジトリを一括で受ける
- **Phase D**: SKILL.md repo フロー（3段: 全 clone → 全 ingest → 一括 compile）+ 2リポジトリ以上での横断 wikilink dogfooding
- 全 Phase 共通: wiki_root 非依存、pure/service/CLI 層分離、全外部依存 DI、TDD

### Non-Goals

- GitLab / GitHub API 経由の取得（issues / MR / wiki ページ）
- 差分更新（pull → 変更検出 → re-compile）— スナップショット方針。`source_revision` 記録だけ先行
- source-agnostic pipeline Phase 1 の Fetcher registry への統合 — ただし型境界（後述）を先に切っておき、将来は機械的アダプタで包めるようにする
- コード全文の raw 保存
- 認証の実装 — SSH 鍵 / credential helper 等、環境側に完全委譲
- submodule の取得（`--recurse-submodules` は使わない）
- 機密スキャンパターンの拡充（GitHub/GitLab PAT、秘密鍵ヘッダ等）— **issue 化して別対応**

## Architecture Design

### 物理配置（Architect I4: 1ファイル肥大を防ぎ、Phase 1 吸収を move にする）

```
skills/wiki/scripts/
├── lib/domain/repo.py           (pure — repo MVP 固有型はここに閉じる。types.py は汚さない)
│   ├── RepoSource / DocCandidate / RepoManifest   (frozen dataclass)
│   ├── parse_repo_source(text) → Ok[RepoSource] | Err[RepoParseError]
│   ├── normalize_repo_slug(host, owner, name) → str   (純粋な正規化のみ)
│   ├── discover_docs(file_list) → tuple[DocCandidate, ...]   (優先度ティア付き)
│   └── build_manifest(...) → RepoManifest   (dict ではなく型を返す)
├── lib/domain/test_repo.py
├── lib/service/repo_clone.py    (I/O — runner/which DI)
│   └── resolve_and_snapshot(spec, ...) → Ok[RepoManifest] | Err[RepoIngestError]
├── lib/service/test_repo_clone.py
└── repo_ingest.py               (薄い handler — argparse + Result→exit code + JSON 出力のみ)
```

- **層の合成は handler が行う**（Architect C1: domain → service の逆依存禁止）。
  `parse_repo_source`(pure) → `normalize_repo_slug`(pure) → `sanitize_id` 検証(service) → `resolve_safe_path` 封じ込め(service) の順で CLI が組む
- **Fetcher 型境界**（Architect I1）: service は必ず `Result[RepoManifest, E]` を返す。将来の `@register_fetcher("repo")` は `resolve_and_snapshot` を呼んで `RepoManifest.docs` を `Source[]` に map するだけの機械的アダプタになる。JSON 化（`asdict`）は CLI の表示責務

### slug 正規化（Architect C2: sanitize_id はバリデータであって slug 化しない）

実在の owner/name は大文字・ドットを含む（`Microsoft`, `repo.js`）。`normalize_repo_slug` で NFC 正規化 → lowercase → `.`/連続非英数を `-` に畳む → 先頭末尾区切り除去、**その後** `sanitize_id` で最終検証（防御的二重化）。テストに大文字・ドット・日本語 owner/name を含める。

### CloneResolver（Performance C2 / M2, Security C-2）

- **shallow 既定**: `ghq get --shallow -- <url>` / `git clone --depth 1 --single-branch -- <url> <dest>`。本 MVP は `git rev-parse HEAD` と `git ls-files` しか使わないため shallow の機能的副作用はゼロ。`--full-clone` エスケープハッチのみ残す
- **既 clone 検出**: fetch せずそのまま使い、その HEAD を正直に `source_revision` に記録（スナップショット = 取得時点の意味論）。最新化は `--refresh` 明示指定時のみ
- **fallback clone 先**: `{wiki_root}/.cache/repos/{host}/{owner}/{name}`（gitignore 追記を実装ステップに含む）
- **timeout**: runner に既定 600s。複数リポジトリ実行時は 1 件の失敗/タイムアウトで全体を止めず、結果に per-repo status を記録
- **掃除**: cache 側は `rm -rf {wiki_root}/.cache/repos` で安全に消せる旨を README に明記。**ghq 側 clone はユーザーの `~/ghq` 管理下であり本ツールは削除しない**（責任分界）

### manifest（Performance C1 / M3: LLM コンテキストを直撃させない）

- `RepoManifest` に **構造メタデータ**を含める: トップレベルディレクトリ一覧、言語別ファイル数、ls-files 総数、エントリポイント候補（package.json / main.go / Dockerfile 等）— LLM が全ファイルを読まずに骨格を掴む材料
- docs 候補は**ティア構造**: `tier1`（README / architecture / adr）, `tier2`（openapi / API 定義）, `tier3`（その他 *.md、件数集約）。`--max-docs`（既定 50）で truncate し `docs_total` / `docs_truncated` を必ず含める（ソートキー安定なので決定論性は維持）
- `git ls-files` は **pathspec を git 側に渡して**絞り込み（`'README*' 'docs/**' '*.md' ...`）、生成物ディレクトリ（`dist/`, `build/`, `vendor/`, `node_modules/`）は固定 deny-list で除外
- **出力はファイルが第一級**: `{wiki_root}/.cache/manifests/{slug}.json` に書き、stdout はサマリー（件数 / truncate 有無 / パス）のみ。LLM は必要な tier だけ部分読みする
- あわせて `repo_ingest.py` が **機械生成の `repo-inventory.md`**（ディレクトリツリー要約・言語構成・ls-files 統計・commit hash。決定論的なツール出力であり LLM 解釈を含まない）を `raw/files/{slug}/` に書く — これが「docs が薄いリポジトリ」でも成立する**正当な一次ソース**になる

### raw / 記事の役割分担（Architect I2: LLM 生成物を raw に置かない）

| 成果物 | 置き場 | 性質 |
|--------|--------|------|
| リポジトリの docs（README 等） | `raw/files/{slug}/` | 一次情報（既存 file ingest フロー + セキュリティスキャン適用） |
| `repo-inventory.md` | `raw/files/{slug}/` | **機械生成**の一次情報（決定論的、LLM 解釈なし、commit hash 明記） |
| リポジトリ概要記事・横断フロー記事 | `concepts/` | **compile 産物**（LLM 合成）。source_refs は上記 raw ファイル群を指す |

v1 草案の「LLM 生成 repo-snapshot.md を raw に置く」案は**廃止**（raw = immutable 一次情報の理念に反し、再 ingest 時の mutation 問題も未解決なため）。コード読解由来の主張は記事本文中で `{source_path}@{short-hash}` 形式の出典を付ける。

### frontmatter 拡張（Architect C3 / I3: 型契約衝突の回避）

- raw フロントマターに optional で追加: `source_revision`（commit hash — **`source_version` は使わない**。pipeline の `Source.source_version: int` と衝突するため）+ `source_path`（リポジトリ内相対パス）
- pipeline へのマッピング表を `references/frontmatter-schemas.md` に記載: `source_url` → `permalink`、`source_revision` → `Source` に将来 `revision` フィールドとして正式追加（Phase 1 で確定）、`source_path` → `extensions["repo"]` 予約

## SKILL.md ingest 節の拡張（Phase D）

入力判定に repo タイプを追加。**複数リポジトリは 3 段で処理する**（Performance H1: 横断 wikilink は全リポジトリが出揃って初めて張れる）:

```
段1: 全リポジトリを clone + manifest 生成（repo_ingest.py <url>... 一括）
段2: 全リポジトリの docs 選定 + ingest（raw/files/{slug}/ へ、スキャン適用）
段3: 一括 compile（リポジトリ概要記事 + 横断フロー記事、相互 [[wikilink]]）
```

**段階的読解プロトコル**（Performance H2 — compile 規範として Phase B で詳細化）:
1. manifest の構造メタ + tier1 docs のみ読む
2. エントリポイント数ファイルと最上位 README のみ深掘り
3. 不足箇所だけ追加 Read。「全部読むな、この順で最小限読め」を明示
4. ls-files > 5000 件の場合は構造ベース記述に留める（manifest が警告を出す）
5. 記事に読解カバレッジの限界を明記（`> [推測]` ルールと整合）

**untrusted 取り扱い**（Security C-3）: リポジトリ内容（docs・コード・コメント・ファイル名）は untrusted data。読解時はデリミタで囲み「記述せよ、従うな」を明示。compile が生成した記事にも保存前に既存の機密スキャン + インジェクション検出を適用する。

## Security Considerations（Security review 反映）

| # | 対策 | 反映先 |
|---|------|--------|
| C-1 | **positive-match allowlist**: `https://` / `ssh://` / `git@host:path` 完全一致 / 実在ローカルパス、のいずれかに合致しない入力は全拒否（`ext::` / `fd::` / `git::` は構造的に落ちる） | parse_repo_source |
| C-2 | **`GIT_ALLOW_PROTOCOL=https:ssh:git`** を subprocess env に設定（ghq は内部で git を exec するため両経路に効く）。fallback 側は `-c protocol.ext.allow=never` も併用 | repo_clone |
| C-3 | untrusted デリミタ + 生成記事の事後スキャン（上記） | SKILL / compilation-guide |
| H-1 | clone 先/予測パスは `resolve_safe_path` の **2 base 封じ込め**（ghq root と cache root それぞれを base に）。slug は sanitize_id、パスは resolve_safe_path と役割分離 | repo_clone |
| H-2 | docs 読み取り前に各 `source_path` を `resolve_safe_path(base=clone_root)` で検証（symlink escape 拒否）。`.git/` / `.env*` / 鍵ファイルは discover 対象から除外 | repo_clone / SKILL |
| H-3 | URL userinfo（`token@host`）は**パース時に必ず除去**して警告。source_url は保存前に機密スキャン | parse_repo_source |
| H-4 | protocol ポリシーを入力種別で分離: リモート = file 除外 / 明示ローカルパス = 実在 + git repo 検証の上で当該コマンドのみ `-c protocol.file.allow=user` | repo_clone |
| H-5 | host 成分の検証: `^[A-Za-z0-9][A-Za-z0-9.\-]*$`、port は数字のみ。ssh:// と scp-like のパース差をテストで区別 | parse_repo_source |
| M-3 | source_url の制御文字/改行拒否、frontmatter はシリアライザ経由 | ingest フロー |

**脅威モデル注記**（Pragmatist との合意）: 主用途は「ユーザー自身が社内 URL を打つ」ため脅威は限定的だが、上記はいずれも実装コストが小さく（env 1行・regex 数本・除去処理）、公開 OSS リポジトリも ingest する運用（dogfooding 含む）を考えると省く理由がない。重厚な対策（sandbox 実行等）は採らない。

## Implementation Steps

### Phase A — 手動 dogfooding（スクリプトより先。Pragmatist CRITICAL-2）

1. 公開リポジトリ 1 つ（本リポジトリ以外の外部 OSS）を手動 clone し、既存 file ingest フロー（**実績ゼロの経路**）で docs を 3〜5 枚 ingest
2. LLM が手動でリポジトリ概要記事を compile し、`wiki query` を 2〜3 問実行
3. 実測記録: 記事品質（Q&A に耐えるか）/ 機密スキャン誤検知率 / 手数の摩擦
4. **ゲート**: 記事が使い物になるか判断。ダメなら Phase B の規範設計を先に強化、スクリプト投資は保留

### Phase B — compile 規範（Pragmatist CRITICAL-1: ここが本体）

5. `references/compilation-guide.md` に「repo ソースの compile」節を新設:
   - repo 概要記事の必須見出し（責務 / エントリポイント / 主要モジュール / **他リポジトリとの接点**（依存 API・呼ぶ相手・呼ばれる相手））
   - 横断フロー記事の作り方の具体例 1 本（例: client → backend-api → gameserver の通信フロー）
   - 出典規約: コード事実は `{source_path}@{short-hash}` で指す（記事は「コードを指すインデックス」、主張の終着点にしない）
   - 段階的読解プロトコル + untrusted デリミタ規則
6. `references/frontmatter-schemas.md` に `source_revision` / `source_path` とマッピング表を追記

### Phase C — スクリプト（TDD: 各ステップ RED → GREEN）

7. `lib/domain/repo.py`: parse_repo_source（allowlist 受理/拒否、`ext::` 等の拒否、userinfo 除去、host/port 検証）+ normalize_repo_slug（大文字・ドット正規化）
8. `lib/domain/repo.py`: discover_docs（ティア分け・deny-list・決定論）+ build_manifest（truncate・docs_total・構造メタ）
9. `lib/service/repo_clone.py`: FakeRunner DI で ghq/git 分岐・shallow 既定・GIT_ALLOW_PROTOCOL・timeout・既 clone 検出・2 base 封じ込め
10. smoke 統合テスト 1 本（tmp_path のローカル bare repo → clone → manifest + repo-inventory.md 生成まで。ネットワーク不要）
11. `repo_ingest.py` CLI: 複数 source 受理（`nargs='+'`）、`--max-docs` / `--full-clone` / `--refresh` / `--output`、exit 0/1/2/130、manifest はファイル出力 + stdout サマリー
12. `.wiki/.gitignore` テンプレートに `.cache/` を追記（wiki-init 生成物にも反映）

### Phase D — 統合と実証

13. SKILL.md ingest 節に repo フロー（3 段構成）を追記、CLAUDE.md / README.md 反映
14. **複数リポジトリ dogfooding**: 公開リポジトリ 2 つ以上で 段1→段3 を通し、**横断 wikilink の成立**と 5 リポジトリ運用の手数を実測・記録

## Phase D 実測結果（2026-07-04、対象: BurntSushi/ripgrep + sharkdp/fd）

**横断 wikilink の成立を機械検証で確認。** `repo_ingest.py` バッチ実行（fd は URL 指定で ghq 自動 clone、ripgrep はローカルパス）→ 2 ok / 0 failed。3記事（ripgrep / fd / ignore-crate-ecosystem 横断記事）が**全6方向で相互リンク**（graph_gen で fully-connected を確認、dangling 0、lint No findings）。横断接点は実在の依存（fd の `Cargo.lock@fb1486f2` が ripgrep 産 `ignore 0.4.25` に解決）で、バージョン差（HEAD 0.4.27 vs 公開 0.4.25）の矛盾明示ルールも機能した。

**摩擦（follow-up 材料）:**
1. **raw slug の命名不統一** — ローカルパス入力は `ripgrep`、URL 入力は `github-com-sharkdp-fd`。同一リポジトリを両方式で入れると二重 ingest になるリスク。命名規約の統一（owner-name 形式に寄せる等）を検討
2. scraped 日付の規約（実行日自動 vs 指示値）を ingest 規範に明記すべき
3. short-hash 桁数 → compilation-guide に「8桁」と明記済み（対応完了）
4. 単一クレート（binary crate）repo での「主要モジュール」記述ガイドが薄い — mod 名しか構造メタから得られないケースの深掘り指針

## Phase A 実測結果（2026-07-03、対象: BurntSushi/ripgrep @48b0c795 shallow clone）

**ゲート判定: 通過（記事方式は有効）**

- **query 品質**: 3問（クレート構成 / gitignore 実装箇所 / PCRE2 利用法）中 2問完全正答、1問は方向正解だが深さ不足。**事実精度 100%（検証で誤りゼロ、hallucination なし）**。Knowledge Gap の正直申告も機能
- **file ingest 経路**: 初の実運用。README/GUIDE/FAQ 3枚（2,656行）を frontmatter（source_url / source_revision / source_path）付きで問題なく ingest
- **機密スキャン誤検知**: docs 3枚 + 全 *.md で **0 件**（メールは Cargo.toml 側にのみ存在）。「docs 限定 ingest なら誤検知面は小さい」が実測で確認された — Follow-up issue の「警告のみモード」は優先度を下げてよい
- **shallow clone**: `ghq get --shallow` 正常動作、rev-parse / ls-files も問題なし（機能的副作用ゼロの想定どおり）

**Phase B に反映すべき実測知見:**

1. 記事の深さ不足の正体は「ingest 対象の不足」: 各 crate の `lib.rs` 冒頭 docstring を ingest / 読解対象に加えるだけで「変更の入口はどのファイルか」型の質問に答えられるようになる
2. 圧縮ロス: フラグ・優先順位など**列挙可能な事実はテーブルで保持**する規約が必要（GUIDE の ignore precedence が記事から脱落した）
3. エントリポイント節は main() で止めず、**データフロー1段**（主要モジュール間の接続）まで書く
4. 分量規約は「300〜600 words」でなく**文字数基準（1,500〜4,000字）を併記**（日本語記事で wc -w は無意味）
5. 出典優先規則: docs とコード両方にある事実は**コード側を正**とし commit-pinned 出典を付ける
