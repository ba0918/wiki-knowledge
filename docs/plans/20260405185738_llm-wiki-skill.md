## 📝 What & Why

Karpathy の「LLM Wiki」コンセプトを Claude Skill として実装する。LLM がソースドキュメントを知識ベース（相互参照付き Markdown Wiki）にコンパイル・メンテナンスし、人間はソースのキュレーションと質問に集中できる仕組みを構築する。先人の実装（pedronauck/karpathy-kb, kfchou/wiki-skills, xoai/sage-wiki 等）の分析結果を反映し、Ingest/Compile 分離、Query→Wiki Promote、Backlink Audit 必須化を組み込む。

## 🎯 Goals

- Claude Skill として既存プロジェクトに導入可能な LLM Wiki 知識ベースを構築する
- Phase 0（MVP）: wiki-init / wiki-ingest / wiki-compile の3スキルで「動く最小のWiki」を実現
- Phase 1: wiki-query / wiki-lint を追加し、4相パイプライン（Ingest→Compile→Query→Lint）を完成
- Phase 2+: RRF ハイブリッド検索、信頼度スコア、ポータル統合をロードマップに沿って段階的に追加
- テスタビリティ・セキュリティ（入力汚染検出・機密マスキング）を初期から組み込む

## 📐 Design

### ディレクトリ構造（プロジェクト側）

Wiki のデータは単一ディレクトリ（デフォルト `.wiki/`）に集約し、既存プロジェクトのルートを汚染しない。
`wiki-init` 時に `--path <dir>` でカスタマイズ可能（例: `wiki/`, `docs/wiki/` 等）。

```
{project-root}/
├── CLAUDE.md                     # Wiki スキーマドキュメント（scope、規約、記事一覧、研究ギャップ）
└── .wiki/                        # デフォルトパス（wiki-init --path で変更可能）
    ├── raw/                      # 不変のソースドキュメント（人間がキュレーション）
    │   ├── articles/             # Web記事、ブログ、論文
    │   │   └── {YYYYMMDD}-{slug}.md
    │   └── files/                # ローカルファイル、PDF等
    ├── concepts/                 # LLM生成 Wiki記事（相互参照付き、フラット）
    │   └── {slug}.md
    ├── outputs/                  # Query 結果・Lint レポート
    │   ├── queries/              # Query回答（promote候補）
    │   └── reports/              # Lint レポート
    ├── schema/                   # 構造定義（静的JSON、人間管理）
    │   ├── page-template.json    # Wikiページのフィールド定義
    │   └── categories.json       # カテゴリ一覧
    ├── index.md                  # 全ページカタログ（カテゴリ別、1行サマリー）
    └── log.md                    # Append-only 操作ログ（grep可能）
```

**パス解決**: 各スキルは CLAUDE.md の `wiki_root` フィールドからベースパスを取得する。CLAUDE.md が見つからない場合は `wiki-init` の実行を促す。

### スキル構成（claude-skills 側）

```
skills/wiki/
├── SKILL.md                      # メインスキル定義（ルーティング）
├── references/
│   ├── architecture.md           # 設計思想・3層構造・4相パイプライン
│   ├── compilation-guide.md      # Compile 時のルール（語調、wikilink密度、出典）
│   ├── frontmatter-schemas.md    # フロントマター定義（wiki/raw/output 各種）
│   ├── lint-procedure.md         # Lint チェック項目と修復フロー
│   └── prompts.md                # LLM プロンプトテンプレート集
├── scripts/
│   └── lint-wiki.py              # 自動Lint（dead link、orphan、missing source）
└── assets/
    ├── wiki-article-template.md  # Wiki記事テンプレート
    ├── index-template.md         # index.md テンプレート
    ├── log-template.md           # log.md テンプレート
    └── claude-md-template.md     # CLAUDE.md テンプレート
```

### 4相パイプライン

```
Phase 1: INGEST（{wiki_root}/raw/ へステージング）
  → ソースを {wiki_root}/raw/ に保存（immutable）
  → フロントマター付与（title, source_url, scraped, tags）
  → 入力汚染検出 + 機密マスキング
  → {wiki_root}/log.md 追記
     ↓
Phase 2: COMPILE（raw/ → {wiki_root}/concepts/ へ記事生成）
  → {wiki_root}/schema/page-template.json + CLAUDE.md 読み込み
  → 既存 {wiki_root}/index.md でオリエンテーション
  → LLM に記事生成指示（出典明記、[[wikilink]] 付き）
  → Backlink Audit（既存記事を grep → 双方向リンク追加）必須
  → {wiki_root}/index.md 更新、CLAUDE.md 更新
  → {wiki_root}/log.md 追記、git commit
     ↓
Phase 3: QUERY（wiki から回答合成）
  → {wiki_root}/index.md → 関連記事特定 → grep/全文ロード
  → [[wikilink]] 付き引用で回答合成
  → {wiki_root}/outputs/queries/ に保存
  → 品質が高ければ {wiki_root}/concepts/ に promote
  → {wiki_root}/log.md 追記
     ↓
Phase 4: LINT（品質チェック + 修復提案）
  → 自動: dead link, orphan, missing source（lint-wiki.py）
  → LLM駆動: 矛盾検出, 陳腐化, カバレッジギャップ, フォーマット違反
  → severity 3段階（🔴 Error / 🟡 Warning / 🔵 Info）
  → 修復は diff 提示 → 人間承認（INFO レベルのみ自動）
  → {wiki_root}/outputs/reports/ にレポート保存
  → {wiki_root}/log.md 追記
     ↓
  back to Phase 1

※ {wiki_root} は CLAUDE.md の wiki_root フィールド値（デフォルト: .wiki）
```

### Wiki ページフロントマター

```yaml
---
title: ページタイトル
type: wiki
source_refs:
  - "raw/articles/20260405-karpathy-llm-wiki.md"  # {wiki_root} からの相対パス
created: 2026-04-05
updated: 2026-04-05
category: concepts
tags: [llm, wiki, knowledge-base]
related:
  - "concepts/three-layer-structure.md"            # {wiki_root} からの相対パス
---
```

### log.md 形式（grep可能、append-only）

```markdown
## [2026-04-05] ingest | karpathy-llm-wiki.md (article)
## [2026-04-05] compile | LLM Wiki Architecture (2847 words, 3 sources)
## [2026-04-05] query | How does Ingest differ from Compile?
## [2026-04-05] promote | Ingest vs Compile Comparison (from query)
## [2026-04-05] lint | 2 dead links, 1 orphan, all fixed → [[2026-04-05-lint]]
```

### Key Points

- **単一ディレクトリ集約**: Wiki 全データを `{wiki_root}`（デフォルト `.wiki/`）に閉じ込め、既存プロジェクト構造を汚染しない。`wiki-init --path` でカスタマイズ可能
- **パス解決**: 各スキルは CLAUDE.md の `wiki_root` フィールドからベースパスを取得。Wiki 内パスは全て `{wiki_root}` からの相対パス
- **Ingest/Compile 分離**: pedronauck パターン。raw/ は immutable、バッチ取り込み→後から一括コンパイル可能
- **Backlink Audit 必須化**: Compile 時の必須ステップ。skip すると wiki が blog に退化する
- **Query→Wiki Promote**: 回答を outputs/ に保存、品質が十分なら concepts/ に昇格。知識の複利的成長
- **CLAUDE.md スキーマ**: プロジェクトルートに配置。LLM が自然に読むスキーマドキュメント。`wiki_root` フィールドで Wiki パスを宣言
- **log.md 監査ログ**: git commit と並行。`grep "^## \[" {wiki_root}/log.md | tail -10` で直近10件
- **セキュリティ**: Ingest 時の入力汚染検出（プロンプトインジェクション）+ 機密データマスキング（正規表現）
- **検索 DI**: Strategy Pattern で grep → BM25 → ハイブリッドを段階的に差し替え可能

### Phase 2+ で検討する技術

- **RRF ハイブリッド検索**: `score = 1/(60+bm25_rank) + 1/(60+vec_rank) + tagBoost + recencyDecay`
- **Ontology グラフ**: 8種リレーション（implements, extends, contradicts 等）+ BFS 探索
- **sqlite-vec**: ファイルベースのベクトル検索（外部DB不要）
- **コンテキストバジェット管理**: ingest 30K / query 63K / lint 123K per batch
