# 月額課金不整合 調査手順ナレッジ

> 「課金したのに使えない」「解約したのに請求された」型の問い合わせを、決済側と利用権側の境界跨ぎで
> 裏付けるツール非依存の調査手順。規範全体は `skills/wiki/references/support-investigation-guide.md`、
> レポートは `skills/wiki/assets/support-report-template.md` に従う。本記事は特定の調査ツールを前提にしない。

## 対象の不整合パターン（閉集合）

問い合わせ文面は無限に揺れるため、文面別でなく不整合パターン別の閉集合で扱う。表現揺れによるギャップ
分裂を防ぐ。

| パターン | 意味 |
|---|---|
| 決済あり・利用権なし | 決済側は成立、利用権が付いていない |
| 決済なし・利用権あり | 決済側に記録がないのに利用権が付いている |
| 表示のみ不一致 | 決済・利用権は一致、画面表示だけがずれている |
| 照合不能 | 決済側と利用権側を突き合わせる共通キーが欠けている |
| 処理中・再照会待ち | settlement window 未経過。確定前で判断保留 |

新しいパターンを見つけたら、閉集合の進化ガバナンス（新カテゴリか既存 variant か）に従って判断し、その
結果を本記事に記録する。

## 共通照合キー

境界跨ぎ調査は「2 システムを検索」でなく「共通照合キー + 基準時刻での突き合わせ」。

| キー | 用途 |
|---|---|
| ユーザー ID | 調査の起点。仮名化してレポートに記す |
| 注文 ID / 購入トークン | 決済側と利用権側を突き合わせる要 |
| プラン ID | どの利用権ルールを当てるかを決める |

対応表（ユーザー ID ↔ 注文 ID / 購入トークン）自体の権限・保持・監査も調査対象。キーが欠ければ「照合不能」
（調査不能タクソノミの「照合キー欠落」）に定型化する。

## 基準時刻タイムライン

決済確定（event-time）→ 利用権反映（processing-time）→ 観測時刻を UTC 正規化して並べ、各時刻に時刻種別
ラベル（event / processing）を付す。「画面で確認できた事実」と「決済サービス内部で確定した事実」を区別する。

### settlement window ゲート（必須）

決済確定から利用権反映までは待ち時間（settlement window）を持つ。**window 未経過なら「処理中・再照会待ち」**
とし、バグ推定へ進まない。起点イベント時刻 + window = 確定期限を観測時刻と比較する。

## 判定根拠となる業務ルール（プレーンテキスト）

以下は判定に使う条件の型（将来 discover で business-rules 記事が生成されたら wikilink 参照に置き換える。
現時点では存在しない記事への wikilink は張らず dead_link を避ける）。

- 利用権付与条件（決済確定 → 利用権発行のトリガーと反映タイミング）
- 猶予期間・保留条件（購入保留・支払い猶予・返金遅延で利用権が一時的にずれる）
- 解約反映条件（解約が次回請求サイクルに反映される境界）

## 調査経路スロット（登録済みツール群への抽象カテゴリ）

経路優先順位（証拠権威 = Source of Truth への近さを第一基準、最小権限を tiebreaker）に従って選択する。
決済の SoT は自社画面表示ではなく決済サービスの権威ある経路である点に注意。

| スロット | 役割 | SoT からの距離の目安 |
|---|---|---|
| 公式 API 系 | 決済確定事実の権威ある取得（App Store / Google Play 等の公式 API） | 高（外部確定事実の SoT） |
| データ照会系 | 利用権レコードの read-only 照会 | 内部の一次データ |
| 画面経由系 | 公式 API が無い自社製決済管理ツールの画面確認（API が生えるまでの橋） | 派生ビュー |

IAP（App Store / Google Play）は公式 API への還元を第一経路とし、2FA 管理コンソールの画面経由運用は
恒常化しない（還元判定が先）。read-only を大原則とし、支払手段など回答に不要な機微情報は取得しない
（取得項目最小化）。PII テーブルは直接触れず調査専用ビューを用いる。

## 確信度と還元

確信度は corroboration の数で判定する（高 = 独立した SoT 級 2 つ以上一致 / 中 = 単一 SoT / 低 = 間接推論のみ）。
決済側と利用権側の 2 つの独立した SoT が一致して初めて「高」。Wiki への還元は匿名化した判断規則の
クラスレベルのみ（最小一般性バーを満たす）。
