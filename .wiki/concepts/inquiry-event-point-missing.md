# イベント pt 未付与 調査手順ナレッジ

> 「対局後にイベントポイントが入らなかった」型の問い合わせを、ユーザー ID から裏付けるツール非依存の
> 調査手順。規範全体は `skills/wiki/references/support-investigation-guide.md`、レポートは
> `skills/wiki/assets/support-report-template.md` に従う。本記事は特定の調査ツールを前提にしない。

## 対象の不整合パターン

「イベント pt が付与されるはずの行動をしたのに、付与されていない」という観測に対する調査。付与「されない」
ことには正当な業務ルール（無効試合・対象外・上限・遅延・取消）による説明がありうるため、観測だけで
バグ断定しないことが核心。判断の 3 層分解（観測事実 / 業務ルール適用 / バグ推定）を様式で分離する。

## 共通照合キー

| キー | 用途 |
|---|---|
| ユーザー ID | 調査の起点。仮名化してレポートに記す |
| 対局 ID（行動 ID） | 付与判定の対象イベントを特定する |
| イベント ID | どのイベントの付与ルールを当てるかを決める |

照合キーが欠ける（対局 ID が特定できない等）場合は、調査不能タクソノミの「照合キー欠落」に定型化する。

## 基準時刻タイムライン

対局終了（event-time）→ 付与処理（processing-time）→ 観測時刻を 1 本の UTC 正規化タイムラインに並べる。
各時刻に時刻種別ラベル（event / processing）を付す。

### settlement window ゲート（必須）

イベント付与は対局終了から確定までに待ち時間（settlement window）を持つ。**window 未経過ならバグ推定へ
進まず「処理中・再照会待ち」とする**。起点イベント時刻 + window = 確定期限を計算し、観測時刻と比較する。
これを怠るのが最大の偽陽性。

## 判定根拠となる業務ルール（プレーンテキスト）

付与「されない」ことを説明しうる業務ルールを、観測事実に当てる。以下は判定に使う条件の型であり、実際の
数値・条件は各サービスの付与仕様に従う（将来 discover で business-rules 記事が生成されたら、この節を
wikilink 参照に置き換える。現時点では存在しない記事への wikilink は張らず dead_link を避ける）。

- 無効試合条件（途中切断・不正検知などで対局が無効化された）
- 付与対象外条件（イベント期間外・対象外モード・対象外ユーザー区分）
- 付与上限条件（1 日/1 イベントあたりの上限に達していた）
- 遅延・取消条件（付与がバッチ確定待ち、または後続処理で取り消された）

いずれのルールでも説明できず、かつ settlement window を経過している残差のみをバグ推定に回す。

## 調査経路スロット（登録済みツール群への抽象カテゴリ）

経路は固定ツールではなくカテゴリのスロットで記述する。選択は経路優先順位（証拠権威 = Source of Truth への
近さを第一基準、最小権限を tiebreaker）に従う。具体ツールは各環境の登録に委ねる。

| スロット | 役割 | SoT からの距離の目安 |
|---|---|---|
| 公式 API 系 | 付与結果の権威ある取得（提供されていれば） | 高（外部確定事実） |
| データ照会系 | 付与レコード・対局結果の read-only 照会 | SoT に近い（内部の一次データ） |
| 画面経由系 | 管理画面での確認（API・直照会が無い場合の橋） | 派生ビュー（表示は一次データの派生でありうる） |

read-only を大原則とし、取得項目最小化・出力上限を守る。

## 接続前にできる設計

実接続（credential・実案件）はユーザー同席ステップで行う。接続前でも次を設計しておく。

- 共通照合キーの確定手順（ユーザー ID からどう対局 ID・イベント ID に辿るか）
- 各付与処理の settlement window の把握
- 確認観点（どのルールを、どの順で当てるか）

## 確信度と還元

確信度は corroboration の数で判定する（高 = 独立した SoT 級 2 つ以上一致 / 中 = 単一 SoT / 低 = 間接推論のみ）。
Wiki への還元は匿名化した判断規則のクラスレベルのみ（個別ケースは還元しない・最小一般性バーを満たす）。
