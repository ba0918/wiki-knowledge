# ユーザーサポート裏付け調査の壁打ち裁定

本ドキュメントは、ユーザーサポートの問い合わせ裏付け調査をツール非依存の調査規範として
確立するための壁打ち裁定を記録したものである。新規の調査手順ナレッジ記事の根拠ソースとなる。

## 出発点

ユーザーサポートは「集合知への質問」であり、問い合わせの頻度・定型性が高い。一度確立した調査手順は
繰り返し再利用され、蓄積するほど回転数が上がる。既存の wiki 基盤ループ（記事参照 → 提示 → 人間承認 →
実行 → 参照ログ → ギャップ検出）が、そのままサポートナレッジの成長エンジンになる。

## 主要な裁定

### 判断の 3 層分解

「pt 入ってないね」という一言は 3 つの判断の畳み込みである。

1. 観測事実 — データ源から読み取れた生の事実
2. 業務ルール適用 — 無効試合・対象外・上限・遅延・取消などの条件を当てる
3. バグ推定 — ルールで説明できない残差を不具合の疑いとする（人間の担当）

観測事実だけを見てバグ推定へ飛ぶのが最大の誤成功。

### レポート 5 分割様式

観測事実 / 適用ルール / 推定原因 / 確信度 / 不足情報。二値回答にしない。「権限不足」「照合キー欠落」
「処理中・再照会待ち」などの調査不能も正式な結果として定型化する。調査不能の集計が、API 昇格・監査ビュー
新設・ツール登録の優先度を測る計器になる。

### 調査経路の優先順位

固定のツールリストではなく、登録済みツール群に対する選択ルールとして定義する。第一基準は証拠権威
（その経路が Source of Truth にどれだけ近いか）。内部 API は内部 DB の派生でありうるため、アクセス手段の
種別と真実源を混同しない。tiebreaker は最小権限アクセス。抽象カテゴリ（公式 API 系 / データ照会系 /
画面経由系）で経路を語り、具体ツールは各環境の登録に委ねる。

### 不整合パターン別の閉集合

調査手順ナレッジは問い合わせ文面別（表現が無限に揺れる）でなく不整合パターン別の閉集合として設計する。
月額課金なら「決済あり・利用権なし / 決済なし・利用権あり / 表示のみ不一致 / 照合不能 / 処理中・再照会待ち」。
表現揺れによるギャップ分裂を防ぐ。

### 境界跨ぎ調査

「2 システムを検索」でなく「共通照合キー + 基準時刻での突き合わせ」。内部ユーザー ID ↔ 注文 ID / 購入
トークンの対応表が照合の要。タイムラインには両境界のイベント時刻と観測時刻を並べ、TZ 正規化と
event-time / processing-time の時刻種別区別を必須とする。

### 誤知識の自己強化への防御

Wiki に還元するのは個別ケースの調査結果でなく匿名化した判断規則のクラスレベルのみ。ケース記録庫化と
検索ノイズを防ぐ。記事を信じて調査を省略した結論の還流は誤りを固定化するため、複数ケースに適用可能な
最小一般性バーを満たす規則のみ還元する。

## スコープの裁定

- 成果物はサポート担当者向けの調査レポートまで。顧客向け回答文には踏み込まない
- 実行のゲートは常に人間承認。LLM は候補提示まで
- ツール非依存が最優先。具体的な調査ツールはユーザーが各環境で登録し、規範本体はそれらへ構造的に依存しない
- 実データ照会は read-only を大原則とし、取得項目最小化・調査専用ビューを出口条件とする
