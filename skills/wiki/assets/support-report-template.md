## 基準時刻タイムライン

全時刻を **UTC に正規化**し、元 TZ を併記する。各時刻に**時刻種別ラベル**（`event` = 事象発生 /
`processing` = システム処理・記録）を付す。event-time と processing-time の混同が誤読の主因。

| 時刻 (UTC) | 元 TZ | 時刻種別 | 出来事 | データ源 (SoT からの距離) |

### settlement window 確認欄（バグ推定の必須ゲート）

非同期処理・バッチ確定を待たずにバグと断定するのが最大の偽陽性。**確定待ち時間（settlement window）の
経過を確認するまでバグ推定へ進まない**。

| 項目 | 値 |
|---|---|
| 起点イベント時刻 (UTC) | |
| settlement window（確定待ち時間の長さ） | 例: 対局終了から最大 N 分 |
| 確定期限 = 起点 + window (UTC) | |
| 観測時刻 (UTC) | |
| window 経過済みか | はい / いいえ（いいえなら結論は「処理中・再照会待ち」でバグではない） |

## 1. 観測事実

データ源から読み取れた**生の事実のみ**。解釈・推定を混ぜない。どの経路（SoT からの距離）で取得したかを記す。

-

## 2. 適用ルール

観測事実に当てた業務ルールと、その適用結果（無効試合・対象外・上限・遅延・取消 などの条件）。

-

## 3. 推定原因

適用ルールで説明できたか。**バグ推定はこの区画にのみ書く**。settlement window 未経過なら「処理中・再照会待ち」
とし、バグ推定はしない。

-

## 4. 確信度: {高 / 中 / 低}

corroboration（独立した裏付け）の数で判定する。

- **高** — 独立した SoT 級の証拠が 2 つ以上一致（corroboration あり）
- **中** — 単一の SoT による裏付け（独立した第 2 の証拠なし）
- **低** — 間接推論のみ（派生ビューの表示・状況証拠に留まる）

## 5. 不足情報 / 調査不能

追加取得できれば確信度が上がる情報。取得できなかった場合は下記 6 分類のいずれかに定型化する。

| 分類 | 該当 | 還元先 |
|---|---|---|
| 権限不足 | | アクセス権設計・調査専用ビュー |
| 照合キー欠落 | | 対応表の整備 |
| 処理中・再照会待ち | | 再照会（settlement window 経過後） |
| 登録ツール経路なし | | ツール登録バックログ（第一級信号 → Gap Detection） |
| SoT 間不一致 | | データ整合性調査・オーナー確認 |
| retention によるデータ消失 | | 保持ポリシー見直し |
